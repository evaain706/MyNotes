
## LCP

- **TanStack Query(React Query)를 통한 데이터 캐싱:**
    - CommunityMain.tsx 등을 보면 `useQuery`에서 `staleTime: 5 * 60 * 1000` (5분)을 설정하여, 한 번 불러온 데이터를 캐싱해 화면 재진입 시 불필요한 네트워크 요청 없이 메인 콘텐츠를 즉시 띄우도록 최적화했습니다. 이는 체감 로딩 속도 향상에 크게 기여합니다.
- **로딩 UI를 통한 빠른 초기 반응 (Skeleton & Spinner):**
    - 데이터를 불러오는 동안 빈 화면을 노출하는 대신 
        
        QuestionCardSkeleton, `<Spinner />` 등을 우선적으로 렌더링해, 사용자 입장에서 페이지가 즉각적으로 응답하고 있음을 인지하게 만들었습니다
    - 이미지 최적화 


## FCP

- **React.lazy와 Suspense를 통한 코드 스플리팅 (Code Splitting):**
    - QuestionCard.tsx에서 `react-markdown` 모듈을 `LazyReactMarkdown`으로 분리하여 렌더링 중입니다.
    - 마크다운 파서처럼 번들 사이즈를 크게 차지하는 라이브러리를 초기 번들(Bundle)에서 분리하고 필요할 때 비동기적으로 로딩함으로써, 초기 Javascript 로드 용량을 줄여 FCP 지표를 크게 개선했습니다.
- **Vite 기반의 빌드 시스템 적용:**
    - 기본적으로 esbuild와 Rollup을 사용하는 Vite 빌드 툴체인을 채택하여 빠른 콜드 스타트와 고도의 트리 쉐이킹(Tree-Shaking)을 통해 초기 전송되는 JS 파일 크기를 최소화하고 있습니다.



## INP

- **useDebounce 커스텀 훅을 통한 API 호출 최적화:**
    - CommunityMain.tsx의 검색 기능에 커스텀 훅인 
        
        useDebounce를 사용(`debounceSearch`)했습니다.
    - 사용자가 타이핑할 때마다 발생하는 상태 변경 및 API 요청을 300ms 지연시킴으로써, 불필요한 연속 렌더링을 막고 메인 스레드 블로킹 타임을 획기적으로 줄였습니다.
- **React.memo 컴포넌트 단위 메모이제이션:**
    - QuestionCard.tsx, 
        
        OptionCard.tsx, 
        
        Card.tsx 컴포넌트 하단에 `React.memo`를 씌워 엑스포트하고 있습니다.
    - 상위 컴포넌트의 상태가 변하더라도, 전달받은 Props가 동일하다면 재렌더링을 방지해 앱의 전반적인 반응성을 높였습니다.
- **React Hook Form 도입:**
    - package.json에 포함된 React Hook Form을 통해 제어 컴포넌트의 단점(입력마다 폼 전체 재렌더링)을 최소화하는 하단 최적화가 이뤄져 있습니다.


## CLS

- **적극적인 Skeleton UI 컨벤션 사용:**
    - QuestionCardSkeleton, `UserStatisticsSkeleton`, `CommunityDetailSkeleton` 등 로딩 상태에 매칭되는 폭넓은 스켈레톤 컴포넌트들이 구축되어 있습니다.
    - 비동기 데이터가 도착하기 전 예약된 공간을 시각적으로 채워두기 때문에 컨텐츠 팝인(Pop-in) 시 레이아웃 변형이 일어나는 것을 완벽하게 방지합니다.
- **고정된 공간 할당 (Fixed Dimensions & Min-height):**
    - `<div className='... min-h-[20rem] min-w-[34rem]'>` 처럼 스켈레톤이나 컨테이너 요소에 고정된 최소 높이/너비를 CSS 클래스로 명시해 두었습니다.
    - 이를 통해 API 데이터 수신 전후로 브라우저가 할당하는 박스 모델의 사이즈가 유지되어, 스크롤이 갑자기 튀는 CLS 문제를 차단합니다.
