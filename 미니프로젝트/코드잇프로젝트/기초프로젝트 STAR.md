## 프로젝트 개요

**K-POP 팬들을 위한 종합 팬덤 플랫폼 사이트 개발**

---

## 1. 프로젝트 초기 협업 환경 구축 및 공통 컴포넌트 개발

### S (Situation)

프로젝트 시작 전부터 다수의 프론트엔드 개발자가 협업하는 구조였으며,  
코드 충돌 및 개발 속도 저하를 방지하기 위한 사전 준비가 필요했다.

### T (Task)

- 프로젝트 시작 전 협업 규칙 정립
    
- 공통으로 사용될 UI 컴포넌트 선제 개발
    
- 팀 전체가 안정적으로 사용할 수 있는 컴포넌트 설계
    

### A (Action)

- 프로젝트 시작 전
    
    - 역할 분담
        
    - Git 브랜치 전략
        
    - 코드 컨벤션
        
    - 디렉토리 구조  
        를 사전에 합의하여 정리
        
- Button, Input, 후원 Form 등 **공통 컴포넌트 개발 담당**
    
- 단독 프론트엔드 프로젝트 경험만 있었기 때문에
    
    - Git 사용 실수를 방지하기 위해 브랜치 전략과 커밋 흐름을 지속적으로 점검하며 개발
        
- 공통 컴포넌트 설계 시
    
    - props 확장성
        
    - 재사용성
        
    - 스타일 커스터마이징 가능 여부  
        를 우선 고려
        
- 코드 리뷰를 통해
    
    - 불필요한 props 제거
        
    - 네이밍 개선
        
    - 컴포넌트 책임 분리  
        등을 반복적으로 개선
        

### R (Result)

- 공통 컴포넌트가 이후 페이지 개발 시 무리 없이 재사용됨
    
- 팀원들이 추가 수정 없이 바로 활용 가능한 수준의 컴포넌트 품질 확보
    
- 협업 환경에서의 Git 사용 경험 및 안정적인 브랜치 관리 역량 확보

## 2. 마이페이지 개발 및 UI/성능 개선

### S (Situation)

마이페이지는

- 즐겨찾기
    
- 선택
    
- 추가
    
- 삭제  
    등 사용자 상호작용이 빈번한 페이지로, UX와 성능 이슈가 동시에 발생할 수 있는 구조였다.
    

### T (Task)

- 사용자 행동이 명확히 인지되는 UI 구현
    
- 불필요한 리렌더링 및 성능 저하 문제 개선
    

### A (Action)

- 삭제 / 추가 / 선택 시
    
    - 상호작용이 시각적으로 명확히 드러나도록 애니메이션 라이브러리 도입
        
- 개발 과정에서 발견한 문제점:
    
    - 상태 변경 시 불필요한 리렌더링
        
    - 다수 요소가 한 번에 제거될 때의 성능 저하
        
    - 컴포넌트 책임이 과도하게 집중된 구조
        
- 해결 방법:
    
    - 컴포넌트 단위 분리로 렌더링 범위 축소
        
    - `useCallback`, `memo`를 활용해 불필요한 렌더링 최소화
        
    - 상태 변경 로직과 UI 로직 분리
        

### R (Result)

- 체감 성능 및 UI 반응성 개선
    
- Lighthouse 기준:
    
    - 성능: **88**
        
    - 접근성: **100**
        
    - 권장사항: **100**
        
- 초기 목표로 설정했던 성능 기준을 상회하는 결과 달성

## 3. 협업 경험 및 개발 역량 향상

### S (Situation)

팀 단위 프론트엔드 협업과 코드 리뷰 경험이 제한적이었던 상태였다.

### T (Task)

- 협업 과정에서의 커뮤니케이션 능력 향상
    
- 코드 리뷰를 통한 코드 품질 개선
    

### A (Action)

- 팀 컨벤션을 지속적으로 의식하며 개발
    
- 공통 컴포넌트 특성상
    
    - “내가 아니라 다른 사람이 쓰기 쉬운 코드”를 기준으로 설계
        
- 팀원들의 코드 리뷰를 통해
    
    - 내가 미처 고려하지 못한 개선 포인트 학습
        
- 다른 팀원의 코드를 리뷰하며
    
    - 코드 가독성
        
    - 구조적 설계 방식  
        에 대한 이해도 향상
        

### R (Result)

- 협업 프로젝트에서의 커뮤니케이션 역량 강화
    
- 타인의 코드를 읽고 피드백할 수 있는 능력 향상
    
- 단순 구현을 넘어 **사용자·성능·협업을 고려한 개발 관점 확보**

## 프로젝트를 통해 얻은 결과

- 공통 컴포넌트 설계에 대한 실전 경험
    
- 렌더링 최적화와 성능 개선에 대한 실질적 적용 경험
    
- Git 기반 협업 및 코드 리뷰 문화에 대한 이해
    
- 사용자 경험과 성능을 동시에 고려하는 프론트엔드 개발 사고방식 정립
