
## 1. 공통 모달 컴포넌트

### S (Situation)

프로젝트 전반에서 모달 사용 빈도가 높았고, 페이지·기능별로 요구되는 UI와 동작 방식이 상이했다.

### T (Task)

재사용 가능하면서도 유연하게 확장 가능한 모달 공통 컴포넌트를 설계해야 했다.

### A (Action)

- 합성 컴포넌트 패턴을 적용하여 `Header / Content / Footer / Trigger` 영역을 분리
    
- 제어형(controlled)·비제어형(uncontrolled) 모달을 선택적으로 사용할 수 있도록 설계
    
- 각 영역에 대해 스타일 커스터마이징이 가능하도록 구조화
    

### R (Result)

- 페이지별 요구사항에 맞게 모달을 조합하여 사용 가능
    
- 중복 구현 없이 다양한 형태의 모달 UI 대응
    
- 모달 관련 로직과 UI 책임을 명확히 분리
    

### T (Takeaway)

- 모달은 단일 컴포넌트보다 **합성 기반 설계**가 확장성과 유지보수 측면에서 더 적합함을 확인

## 2. 아바타 공통 컴포넌트

### S (Situation)

사용자 프로필 이미지를 여러 화면에서 반복적으로 렌더링해야 했다.

### T (Task)

이미지 로딩 실패 상황까지 고려한 안정적인 아바타 컴포넌트를 구현해야 했다.

### A (Action)

- 이미지 렌더링 전용 컴포넌트로 역할을 한정
    
- 이미지 로드 에러 발생 시 기본 이미지로 대체하는 방어 로직 추가
    

### R (Result)

- 네트워크·이미지 오류 상황에서도 UI 깨짐 없이 안정적인 렌더링 보장
    
- 프로필 이미지 처리 로직을 단일 컴포넌트로 통합
    

### T (Takeaway)

- 반복적으로 사용되는 UI는 **에러 케이스까지 포함해 공통화**하는 것이 유지보수에 유리
    

---

## 3. 체험 상세페이지

### S (Situation)

체험 예약의 중심이 되는 페이지로, 사용자 권한에 따라 전혀 다른 UI와 기능이 필요했다.

### T (Task)

체험 상세 정보 표시와 동시에 예약/수정/삭제 분기 로직을 명확히 구현해야 했다.

### A (Action)

- `useParams`로 체험 ID 추출 후 `useQuery`로 상세 데이터 요청
    
- 로그인 여부 및 체험 작성자 여부에 따라 예약 UI 또는 수정/삭제 UI 분기
    
- 상세 데이터를 타이틀, 이미지, 리뷰, 지도, 예약 인터페이스 등으로 컴포넌트 분리
    

### R (Result)

- 사용자 상태에 따른 명확한 UX 분기 구현
    
- 데이터 의존성이 높은 페이지를 컴포넌트 단위로 안정적으로 구성
    

### T (Takeaway)

- 권한 분기가 있는 페이지일수록 **초기 설계 단계에서 UI 흐름을 명확히 정의**하는 것이 중요

## 4. 예약 인터페이스 (캘린더 기반)

### S (Situation)

예약 캘린더에서 이전·다음 달 날짜가 함께 노출되는 구조로 인해 데이터 불일치 문제가 발생했다.

### T (Task)

캘린더 UI에 필요한 예약 가능 일자를 정확히 표시해야 했다.

### A (Action)

- 현재 달 기준으로 이전/현재/다음 달 예약 가능 일자를 각각 요청
    
- `Promise.allSettled`를 사용해 부분 실패에도 성공 데이터만 병합
    
- Day.js를 사용해 날짜 계산 및 캘린더 렌더링 처리
    

### R (Result)

- 월 이동 없이도 예약 가능한 날짜를 직관적으로 확인 가능
    
- 네트워크 실패에도 예약 UI 전체가 중단되지 않는 구조 확보
    

### T (Takeaway)

- 캘린더 UI는 **시각적 범위와 데이터 범위가 다를 수 있음**을 고려해야 함

## 5. 예약 상태 관리 (Zustand)

### S (Situation)

예약 인터페이스가 여러 하위 컴포넌트로 구성되며 상태 의존성이 높았다.

### T (Task)

날짜·시간·인원·예약 가능 여부 상태를 효율적으로 관리해야 했다.

### A (Action)

- 예약 관련 상태를 Zustand 스토어로 중앙 관리
    
- 각 컴포넌트에서 필요한 상태만 선택 구독하도록 설계
    

### R (Result)

- props drilling 없이 상태 공유 가능
    
- 불필요한 리렌더링 최소화
    
- 예약 플로우 로직 가독성 향상
    

### T (Takeaway)

- 특정 기능 단위로 강하게 결합된 상태는 **경량 전역 상태 관리가 더 적합**

## 6. 체험 등록 페이지

### S (Situation)

체험 생성 시 입력 항목이 많고, 유효성 검증 로직이 복잡했다.

### T (Task)

사용자 입력 실수를 최소화하고 안정적인 등록 플로우를 구현해야 했다.

### A (Action)

- 입력 영역을 컴포넌트 단위로 분리
    
- 커스텀 훅으로 상태 및 검증 로직 분리
    
- 날짜·시간 유효성 검사 및 이미지 업로드 실패 방어 로직 구현
    

### R (Result)

- 모든 필수 입력값 충족 시에만 등록 가능
    
- 에러 상황을 토스트로 즉각 피드백
    
- 등록 성공 시 상세페이지로 즉시 이동
    

### T (Takeaway)

- 폼 로직은 **UI와 분리된 커스텀 훅 구조**가 재사용성과 테스트에 유리
    

---

## 7. 체험 수정 페이지

### S (Situation)

등록과 유사하지만 기존 데이터를 기반으로 동작해야 했다.

### T (Task)

등록 페이지 구조를 최대한 재사용하면서 수정 전용 로직을 구현해야 했다.

### A (Action)

- `useQuery`로 기존 체험 데이터 로드 후 default value로 세팅
    
- 등록과 동일한 컴포넌트 사용, 훅만 수정 전용으로 분리
    
- 수정 성공 시 쿼리 무효화 후 상세페이지 이동
    

### R (Result)

- 코드 중복 없이 등록/수정 페이지 구성
    
- 최신 데이터가 반영된 상세페이지 보장
    

### T (Takeaway)

- **UI 재사용 + 로직 분리**는 CRUD 페이지 설계의 핵심 패턴임을 확인
